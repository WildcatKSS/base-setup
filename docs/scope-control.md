# Scope Control

## Purpose

Prevent Claude Code and the founder from building too much before validation.

## Scope Rules

A feature may enter MVP only if:
1. It solves the core customer problem
2. It supports payment or activation
3. It is needed for first 10 customers
4. It does not materially harm the 60-day build target
5. It does not materially harm unit economics

## Do-Not-Build List

Until explicitly approved, do not build:
- Admin dashboards beyond essentials
- Complex analytics
- Multi-language support
- Mobile apps
- Public API
- Advanced roles
- Team collaboration features
- Marketplace functionality
- Complex onboarding flows
- Custom integrations
- White-labeling
- AI agents with unbounded API cost

## Lean Scope Control

Before adding a feature, answer:

- Can another feature be removed?
- Can existing functionality absorb this requirement?
- Can the requirement be simplified?
- Can this be solved with documentation or process?
- Is this required for the first paying customers?
- Does this improve activation, payment, retention or delivery of the core promise?

If not, classify as `Won't Have` or `Later`.
